# 디자인 명세 v2 구현 (Maison de Balance)

## 너의 역할

너는 이미 동작 중인 이 레포(Next.js App Router + Supabase + Vercel) 위에 **프론트엔드 디자인 명세서 v2를 구현**한다.
새로 만드는 것이 아니라, **기존 구조·데이터·보안 위에 디자인을 입히는** 작업이다. 본질은 **스타일링·UI 구현**이다.

## 작업 대상 Phase

`$ARGUMENTS`가 비어 있으면 현재 브랜치·코드 상태를 보고 **지금 진행할 Phase를 판단해 제안**한 뒤, 사용자 확인을 받고 시작한다.
`$ARGUMENTS`에 Phase 번호(0~5)가 있으면 **그 Phase만** 수행한다.
**한 번에 하나의 Phase만** 수행하고, 끝나면 멈춰 보고한다. 승인 없이 다음 Phase로 넘어가지 않는다.

## 입력 자료 (작업 시작 시 반드시 읽는다)

1. **디자인 명세서 v2** — `docs/Maison_de_Balance_Design_Spec_v2.md` (구현 계약)
2. (선택) **목업** — `docs/Maison_de_Balance_Mockups_v2.html` (시각 참고)
3. **실제 코드베이스** — 직접 읽는 현재 레포

우선순위: **실제 코드 > 명세서 > 목업.**
- 명세 §0(구조 바인딩) 값은 코드 실측 스냅샷이다. 항상 **현재 코드의 실제값**에 바인딩하고, 명세 §0과 코드가 다르면 **코드를 따르고** 불일치를 보고한다(코드를 명세에 맞추지 마라).
- 목업과 명세가 다르면 **명세 우선**(목업은 삽화).

## 절대 규칙 (위반 금지)

1. **디자인이 코드 구조에 바인딩된다. 그 반대가 아니다.** 디자인을 맞추려고 스키마·마이그레이션·RLS·status enum·섹션 키·필드명·조건부 로직을 **변경하지 마라.** 충돌하면 멈추고 보고한다.
2. **보안 레이어 불변.** 미들웨어 role 검사, 서버 액션·쿼리의 role 재확인, RLS, 서버사이드 마스킹, RRN 비복호화 — **건드리지 않는다.**
3. **토큰만 사용.** 색·간격·타이포·반경은 명세 토큰값으로만. 임의 HEX·px 금지. 기존 스타일링 방식을 먼저 파악해 그 체계 안에서 정의·사용한다(새 스타일링 패러다임 도입 금지).
4. **비노출 원칙.** 위험신호·관리자 메모는 **환자 화면 절대 노출 금지.** 목록에 연락처 원문·RRN·주소 금지. RRN은 `rrn_mask`만. 상세 라우트는 **`/admin/responses/[id]`** (레거시 `/admin/[id]` 사용·복원 금지).
5. **기존 컴포넌트·라우트 재사용·확장.** 평행 구조 신규 생성 금지(드리프트 방지).
6. **단계별 진행 + 정지.** 한 Phase 끝나면 멈춰 보고하고 승인 대기.
7. **자동 커밋·푸시·배포 금지.** 변경은 제안·구현하되 커밋/배포 시점은 사용자가 정한다. main 직접 수정 금지, 작업 브랜치(`feat/design-v2`)에서 진행.
8. **추측 금지.** 구조가 불확실하면 코드를 더 읽거나 멈추고 묻는다.

### 범위 밖 (명세 §0-1 백로그 — 보고만, 구현·수정 금지)

- `admin_memo` UPDATE grant 이슈 → DB 작업. 단, **저장 실패 UX(§G-5)는 Phase 4에서 반드시 구현**.
- `anon grant` 잔존, `health_score/pdf_generated/score_components` 미구현 → 대시보드에서 **"예정" 영역**으로 자리만 예약, 값 미표시.

## Phase별 작업

### Phase 0 — 그라운딩 & 준비
- 명세서 v2 전체 읽기.
- 현재 **스타일링 방식**과 **기존 토큰·글로벌 CSS·테마** 위치 파악.
- 명세 §0 바인딩 값(status enum·전이, 섹션 키·순서 2종, 마스킹 포맷, 수정 가능 컬럼, 조건부 게이트, BMI 구간, risk_flags 구조)을 **현재 코드와 대조**.
- 작업 브랜치 생성.
- **코드 변경 없음**(브랜치 제외). 보고: 스타일링 방식 / 기존 토큰 위치 / 명세 §0 vs 코드 불일치 목록 / 구현 계획 개요.

### Phase 1 — 디자인 토큰 & 타이포 기반
- §C·§D 토큰을 기존 체계에 정의: 색(11+4 semantic), 간격(4/8/12/16/24/32/48), 반경(input·button 10/card 12–16/badge 999), 타입 스케일.
- 폰트: Pretendard(본문·UI), Noto Serif KR(제목), 시스템 폴백. 입력 16px 고정.
- 글로벌 base + focus ring 골드 2px.
- **완료 기준:** 이후 작업이 토큰만으로 가능. 하드코딩 색·px 0.

### Phase 2 — 공통 프리미티브
- Button(주=차콜/보조=고스트), Input/Textarea, Card, Badge(status 4값 + severity 3등급), Accordion, Progress bar(숫자 없음), Sticky bar, focus/disabled.
- severity: high=danger-soft / medium=warning-soft / low=info-soft, **없음→medium 폴백**.
- status: 완료=success-soft / 진행=info-soft / 보류·취소=subtle.
- **완료 기준:** §H 가이드와 일치하는 재사용 프리미티브.

### Phase 3 — 환자 설문
- 셸: 1화면=1섹션, 15섹션→6챕터(§F-1), 진행바(숫자 없음), 하단 sticky(다음=차콜 채움).
- 문항: short/long_text, number, time, single/multiple_choice, yes_no, scale_grid(모바일 세로 분해).
- **조건부 게이트 코드 실제 조건 바인딩:** 여성 `basic_sex=="여성"`, 출산 `basic_birth_count!="없음"`. 미해당 렌더 제외 + 맥락 안내 1줄.
- 검증/오류 톤(§F-5·§I): 부드럽게, 빨강·공포 금지.
- intro·동의·완료 화면.
- **위험신호·관리자 메모 노출 금지.**
- **완료 기준:** 전 섹션 작성 흐름 명세대로(제출 로직은 기존 재사용, UI만 교체).

### Phase 4 — Admin Dashboard
- 로그인(§G-1): 계정 존재 암시 없는 오류 톤.
- 목록(§G-2): 마스킹(`010-****-5678`), status 틴트 badge, 위험신호 개수 badge(**high+medium만 집계**, high 있으면 burgundy·없으면 amber, low 미집계), memo 유무, BMI 라벨(§0 구간), 모바일 카드 전환. 행→`/admin/responses/[id]`.
- 상세(§G-4): sticky 요약 → 위험신호(high→medium→low 정렬) → §0 `DASHBOARD_SECTION_ORDER` 순서 → 작업영역. accordion 핵심 오픈. raw answer는 **질문–답변 카드**(JSON 금지). RRN은 `rrn_mask`만.
- 작업영역(§G-5): 수정 가능 필드 **status·admin_memo 뿐**. status는 **전이 규칙(허용값만 노출)**, 명시 저장, "변경됨·미저장", 성공(sage)/**실패 UX 필수**, "관리자 전용" 라벨.
- **완료 기준:** 상세에서 10–30초 핵심 파악 가능. 마스킹·비노출 100%.

### Phase 5 — 상태 · 엣지 · 접근성 마감
- loading(스켈레톤)/empty/error.
- severity 폴백·위험신호 0개·low만 케이스.
- 접근성(§K): 터치 44px+, 키보드·focus, 대비 AA, 색 의존 금지, 긴 선택지 줄바꿈, table→card 반응형.
- **완료 기준:** §K 충족.

## 각 Phase 종료 시 보고 (필수)

1. 변경 파일 + 요약
2. 발견한 **명세 vs 코드 불일치 / 결정 필요 항목** (없으면 "없음")
3. 다음 Phase 제안 + **승인 대기**

## 충돌 처리

- 불일치는 그 자리에서 임의 결정하지 말고 보고 목록에 적는다.
- 디자인을 위해 구조·보안 변경이 "필요해 보이면" 그것이 **멈추고 보고할 신호**다. 변경하지 마라.
- 명세에 없는 미세 디테일은 §B·§L 원칙 안에서 보수적으로 채우되, 규칙급 결정은 보고한다.

시작 전에 **절대 규칙**(특히 1·2·6·7)을 다시 확인하고, 지정된(또는 판단한) Phase **하나만** 수행한다.
