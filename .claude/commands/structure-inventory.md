# 구조 인벤토리 추출 (설문 + 관리자 대시보드)

## 너의 역할

너는 이 레포지토리(Next.js App Router + Supabase + Vercel)의 **실제 구현 구조를 조사해서 보고**하는 역할이다.
지금은 구현·수정 단계가 **아니다**. 프론트엔드 디자인 명세서에 들어갈 "구조 의존 값"을 코드/DB의 실제값으로 확정하기 위한 **인벤토리 추출**만 한다.

대상은 두 영역 모두다.
- **환자 설문(patient survey)** — 환자가 작성하는 초진 설문
- **관리자 대시보드(admin dashboard)** — 내부 staff가 응답을 검토·관리

## 작업 범위

`$ARGUMENTS`가 비어 있으면 아래 0~6 **전체**를 조사한다.
`$ARGUMENTS`에 다음 키워드가 있으면 해당 범위만 조사한다(구조가 일부만 바뀌었을 때의 빠른 재실행용).
- `survey` → 0, 1, 2, 3 (+ 관련 보안)
- `admin` → 0, 1, 2, 4 (+ 관련 보안)
- `security` → 0, 5
- `schema` → 0, 1

범위를 한정하더라도 마지막 `## 명세 영향 요약`은 **항상 포함**한다.

## 절대 규칙 (위반 금지)

1. **읽기 전용.** 코드·스키마·RLS·마이그레이션·환경설정·DB를 **절대 수정하지 마라.** 파일 생성/편집/삭제, ALTER, 마이그레이션 생성 금지.
2. **추측 금지.** 코드·DB에 **실제로 존재하는 것만** 보고한다. 못 찾으면 반드시 `확인 안 됨`이라고 명시한다. 그럴듯하게 메우지 마라.
3. **일반 지식·기억으로 값을 채우지 마라.** 예: status enum 값을 "알 것 같다"고 적지 말고, 코드 또는 DB에서 실제로 확인한 값만 적는다.
4. **모든 항목에 출처를 붙여라.** `파일경로:라인` 형식. 여러 곳이면 모두.
5. **개인정보 실제 값 출력 금지.** 구조 파악은 정보스키마/정책 조회(`information_schema`, `pg_policies` 등) 우선. 실제 응답 행을 select 해야만 형태 확인이 가능한 경우에도 **개인정보 값은 출력하지 말고**(이름/연락처/주민번호 등) 키 구조만 보고한다. 필요하면 `limit 1` + 값 마스킹.
6. **불확실하면 추가 조사하거나 `확인 안 됨`으로 남겨라.** 한 항목을 그럴듯하게 채우는 것보다 비워두는 것이 안전하다.

## 조사 항목

### 0. 프로젝트 지형
- 스택/주요 버전(Next.js, Supabase 클라이언트 등)
- App Router 디렉토리 구조 요약(설문 영역 vs 관리자 영역 파일 위치)
- 스타일링 방식(Tailwind / CSS Module / inline / 기타)
- 기존 디자인 토큰·테마·글로벌 CSS 존재 여부와 위치

### 1. DB 스키마 (Supabase)
- 이 시스템이 사용하는 **모든 관련 테이블**: 컬럼명 / 타입 / nullable / default / 제약조건
- 민감·암호화 컬럼 식별 (예: `rrn_encrypted`, `rrn_mask` 등 — 실제 컬럼명 확인)
- enum / check constraint 전체, **특히 status·gender 관련**
- 마이그레이션 파일 목록과 각 파일이 생성한 것 요약 (`submitted_at`, `health_score`, `pdf_generated` 등 추가 컬럼 포함)

### 2. 상태(status) 워크플로
- **status enum의 실제 전체 값** — 코드와 DB 양쪽에서 확인. 불일치하면 둘 다 보고.
- 상태 전이 규칙(`allowedNextStatus` 등)이 정의된 위치와 허용 전이 매핑
- 제출 시 기본(default) status
- status별로 코드에서 분기되는 동작이 있으면 요약

### 3. 환자 설문 구조
- **섹션 키와 순서**가 정의된 위치(배열/스키마/config 파일). 실제 키 목록과 순서 그대로.
- 각 섹션의 **문항 정의**: `key`, `type`(short_text/long_text/number/time/single_choice/multiple_choice/yes_no/scale_grid/기타), `label`, 필수 여부, 옵션 목록, helper text
- **조건부 로직**: 여성 섹션 노출 게이트(어떤 필드 = 어떤 값), 출산 관련 하위 문항 게이트, 그 외 조건부 — 정의 위치와 표현 방식
- **`raw_answers` 저장 형태**: 키 구조가 섹션.문항인지/평면인지/중첩인지, 실제 키 경로 예시(값 제외)
- 필드별 검증 규칙(숫자 범위, 시간 형식, 필수 등)이 코드에 있으면 위치와 내용

### 4. 관리자 대시보드 구조
- **목록 화면에 노출되는 필드** / **상세 화면에 노출되는 필드** 각각
- 상세 표시 **순서가 코드에 정의**돼 있는지(있다면 위치와 순서 그대로)
- **마스킹**: 어디서(서버/쿼리/유틸) 어떤 포맷으로 마스킹되는가 — phone, rrn 각각. **클라이언트가 실제 받는 형태**(마스킹된 문자열 패턴)
- **risk_flags / 위험신호**: 계산값인지 저장값인지, 스키마(항목 구조 — key/label/severity 등), severity 단계, 임계값(threshold) 정의 위치
- **BMI / health_score**: 계산 위치(클라이언트/서버/DB), 저장 여부
- 관리자가 **수정 가능한 컬럼**(코드 기준) — 5번 RLS와 대조

### 5. 보안 경계 (읽기 전용 확인, 변경 절대 금지)
- 테이블 **RLS 정책**: SELECT / INSERT / UPDATE 각각 — 대상 role, 허용 컬럼
- 미들웨어 role/세션 검사 위치, 서버 액션·쿼리 함수의 role 재확인 위치
- 클라이언트 도달 전에 strip/mask 되는 필드 목록
- `rrn_encrypted` **복호화 경로 존재 여부**(없어야 정상). 있으면 위치를 명확히 보고.

### 6. 라우팅 / 컴포넌트
- 기존 라우트: 설문, admin login, admin list, admin detail (실제 경로)
- 기존 컴포넌트 목록과 스타일 위치(재스타일 대상 파악용)

## 출력 형식

- **단일 마크다운 보고서.** 위 0~6 섹션 헤더를 그대로 사용(범위 한정 시 해당 섹션만).
- 각 항목은 `값 + 출처(파일:라인)` 형식. 못 찾은 것은 `확인 안 됨`.
- 개인정보 실제 값은 절대 출력하지 않는다.
- 마지막에 **`## 명세 영향 요약`** 섹션을 추가하고, 이 인벤토리로 확정 가능한 디자인 명세 placeholder를 매핑한다. 최소 다음을 다룰 것:
  - status enum 확정값
  - 설문 섹션 키 / 순서 확정값
  - 마스킹 포맷(phone, rrn) 확정값
  - 관리자 수정 가능 컬럼 확정값
  - risk_flags 구조 확정값
  - 조건부 게이트(여성/출산 등) 조건 확정값
  - 그 외 명세와 코드가 **어긋난 지점**이 있으면 별도로 표시(`⚠️ 명세 vs 코드 불일치`).

작업을 시작하기 전에, 위 절대 규칙(특히 읽기 전용·추측 금지·PII 비출력)을 다시 확인하고 그대로 따른다.
