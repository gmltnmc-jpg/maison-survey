# Claude Code 작업 지시서 — Maison de Balance 프론트엔드 디자인 구현

## 너의 역할

너는 이미 동작 중인 이 레포(Next.js App Router + Supabase + Vercel) 위에 **프론트엔드 디자인 명세서 v2를 구현**한다.
새로 만드는 것이 아니라, **기존 구조·데이터·보안 위에 디자인을 입히는** 작업이다.

이번 작업의 본질: **스타일링·UI 구현.** 구조(스키마·라우트·필드·status·섹션 키)와 보안(RLS·미들웨어·서버액션·마스킹)은 **이미 확정된 전제**이며, 디자인은 거기에 바인딩될 뿐이다.

---

## 입력 자료

1. **디자인 명세서 v2** — `docs/Maison_de_Balance_Design_Spec_v2.md` (이 작업의 **구현 계약**)
2. (선택) **목업 풀세트** — `docs/Maison_de_Balance_Mockups_v2.html` (시각 참고용)
3. **실제 코드베이스** — 너가 직접 읽는 현재 레포

우선순위: **실제 코드 > 명세서 > 목업.**
- 명세서 §0(구조 바인딩)의 값은 코드 실측 스냅샷이다. 구현 시 항상 **현재 코드의 실제값**을 기준으로 바인딩한다. 명세 §0 값과 코드가 다르면 **코드를 따르고**, 불일치를 보고한다(코드를 명세에 맞추려 바꾸지 마라).
- 목업과 명세가 다르면 **명세가 우선**(목업은 삽화).

---

## 절대 규칙 (위반 금지)

1. **디자인이 코드 구조에 바인딩된다. 그 반대가 아니다.** 디자인을 맞추려고 **스키마·마이그레이션·RLS·status enum·섹션 키·필드명·조건부 로직을 변경하지 마라.** 충돌하면 멈추고 보고한다.
2. **보안 레이어 불변.** 미들웨어 role 검사, 서버 액션·쿼리의 role 재확인, RLS 정책, 서버사이드 마스킹, RRN 비복호화 — **건드리지 않는다.** 이번 작업은 UI 구현이다.
3. **토큰만 사용.** 색·간격·타이포·반경은 명세 토큰값으로만. 임의 HEX·임의 px 금지. (기존 스타일링 방식을 먼저 파악해 그 체계 안에서 토큰을 정의·사용한다 — 새 스타일링 패러다임 도입 금지.)
4. **명세의 비노출 원칙 준수.** 위험신호·관리자 메모는 **환자 화면에 절대 노출 금지.** 목록에 연락처 원문·RRN·주소 노출 금지. RRN은 `rrn_mask`만. 상세 라우트는 **`/admin/responses/[id]`** (레거시 `/admin/[id]` 사용·복원 금지).
5. **기존 컴포넌트·라우트를 재사용·확장**한다. 평행 구조를 새로 만들지 말 것(드리프트 방지).
6. **단계별로 진행하고, 각 단계 끝에서 멈춰 보고한다.** 한 번에 전부 구현하지 마라(아래 실행 방식).
7. **자동 커밋·푸시·배포 금지.** 변경은 제안·구현하되, 커밋/배포 시점은 사용자가 결정한다.
8. **추측 금지.** 구조가 불확실하면 코드를 더 읽거나 멈추고 묻는다. 그럴듯하게 메우지 마라.

---

## 실행 방식

- **작업 브랜치**에서 진행한다(예: `feat/design-v2`). main 직접 수정 금지.
- 아래 **Phase 0 → 5 순서**로 진행한다. 각 Phase 종료 시:
  1. 무엇을 바꿨는지 **변경 파일·요약** 보고
  2. 발견한 **명세 vs 코드 불일치 / 결정 필요 항목** 목록(있으면)
  3. 다음 Phase로 갈지 **사용자 승인 대기** (승인 없이 다음 Phase로 넘어가지 않는다)
- 명세 §0-1 백로그는 이번 작업 범위가 **아니다**(보고만, 구현/수정 금지):
  - `admin_memo` UPDATE grant 이슈 → DB 작업. 단, **저장 실패 UX(§G-5)는 이번에 반드시 구현**한다.
  - `anon grant` 잔존, `health_score/pdf_generated/score_components` 미구현 → 대시보드에서 **"예정" 영역**으로 자리만 예약, 값 미표시.

## Phase 0 — 그라운딩 & 준비

- 명세서 v2 전체를 읽는다.
- 현재 **스타일링 방식**(Tailwind / CSS Module / 기타)과 **기존 디자인 토큰·글로벌 CSS·테마** 위치를 파악한다.
- 명세 §0 구조 바인딩 값(status enum·전이, 섹션 키·순서 2종, 마스킹 포맷, 수정 가능 컬럼, 조건부 게이트, BMI 구간, risk_flags 구조)을 **현재 코드와 대조**한다.
- 작업 브랜치 생성.
- **보고:** 스타일링 방식 / 기존 토큰 위치 / 명세 §0과 코드의 불일치 목록 / 구현 계획 개요.
- ⚠️ 이 Phase에서는 코드 변경 없음(브랜치 생성 제외).

## Phase 1 — 디자인 토큰 & 타이포 기반

- 명세 §C·§D 토큰을 기존 스타일링 체계에 정의: 색(11+5), 간격 스케일(4/8/12/16/24/32/48), 반경(input·button 10/card 12–16/badge 999), 타입 스케일.
- 폰트 로딩: Pretendard(본문·UI), Noto Serif KR(제목), 시스템 폴백. 입력 폰트 16px 고정.
- 글로벌 base(배경 `--color-bg`, 텍스트 `--color-text`, focus ring 골드 2px).
- **완료 기준:** 모든 후속 작업이 토큰만으로 가능한 상태. 하드코딩 색·px 0.

## Phase 2 — 공통 프리미티브

- Button(주=차콜 채움/보조=고스트), Input/Textarea, Card, Badge(status 4값 틴트 + severity 3등급 틴트), Accordion, Progress bar(숫자 없음), Sticky bar, focus/disabled 상태.
- severity 색 매핑: high=danger-soft, medium=warning-soft, low=info-soft. **severity 없음 → medium 폴백**.
- status 색: 완료=success-soft, 진행(신규·예정)=info-soft, 보류·취소=subtle.
- **완료 기준:** 명세 §H 컴포넌트 가이드와 일치하는 재사용 프리미티브 세트.

## Phase 3 — 환자 설문

- 셸: 1화면=1섹션, 15섹션 → 6챕터(§F-1), 진행바(숫자 없음·챕터명), 하단 sticky 이전/다음(다음=차콜 채움).
- 문항 컴포넌트: short_text/long_text/number/time/single_choice/multiple_choice/yes_no/scale_grid(모바일 세로 분해)(§F-3).
- **조건부 게이트는 코드 실제 조건에 바인딩:** 여성 `basic_sex=="여성"`, 출산 `basic_birth_count!="없음"`. 미해당 렌더 제외, 등장 시 맥락 안내 1줄.
- 검증/오류 톤(§F-5, §I): 부드러운 문구, 빨강·공포 금지.
- intro·동의·완료 화면.
- **위험신호·관리자 메모는 이 영역 어디에도 노출하지 않는다.**
- **완료 기준:** 모바일에서 전 섹션 작성 흐름이 명세대로 동작(데이터 제출 로직은 기존 것 재사용, UI만 교체).

## Phase 4 — Admin Dashboard

- 로그인(§G-1): 계정 존재 암시 없는 오류 톤.
- 목록(§G-2): 컬럼·마스킹(`010-****-5678`), status 틴트 badge, 위험신호 개수 badge(**high+medium만 집계**, high 있으면 burgundy·없으면 amber, low 미집계), memo 유무, BMI 라벨(§0 구간), 모바일 카드 전환. 행 클릭 → `/admin/responses/[id]`.
- 상세(§G-4): sticky 요약 바 → 위험신호 요약(severity 정렬 high→medium→low) → §0 `DASHBOARD_SECTION_ORDER` 순서 섹션 → 관리자 작업영역. accordion 핵심 오픈. raw answer는 **질문–답변 카드**(JSON 노출 금지). RRN은 `rrn_mask`만.
- 작업영역(§G-5): **수정 가능 필드는 status·admin_memo 뿐.** status 변경은 **전이 규칙(현재 상태에서 허용된 값만 노출)**, 즉시 저장 금지·명시 저장, "변경됨·미저장" 표시, 저장 성공(sage)/**실패 UX 필수**, "관리자 전용" 라벨.
- **완료 기준:** 상담자가 상세에서 10–30초 내 핵심 파악 가능. 마스킹·비노출 원칙 100% 준수.

## Phase 5 — 상태 · 엣지 · 접근성 마감

- loading(스켈레톤)/empty/error 상태(§I).
- severity 폴백 동작 확인, 위험신호 0개·low만 케이스.
- 접근성(§K): 터치 44px+, 키보드 이동·focus 가시성, 대비 AA, 색 의존 금지, 긴 선택지 줄바꿈, Admin table→card 반응형.
- **완료 기준:** 명세 §K 체크리스트 충족.

## 충돌·결정 처리 규칙

- 명세 vs 코드 불일치를 발견하면 **그 자리에서 임의 결정하지 말고** 목록에 적어 해당 Phase 보고에 포함한다.
- 디자인을 위해 구조·보안 변경이 "필요해 보이면" 그것이 바로 **멈추고 보고할 신호**다. 변경하지 마라.
- 명세에 없는 UI 디테일(미세 간격·문구 등)은 명세 원칙(§B·§L) 안에서 보수적으로 채우되, 새 규칙급 결정은 보고한다.

## 최종 완료 기준

- 모든 Phase가 명세 v2에 부합하고, 구조·보안 레이어는 **단 한 줄도 변경되지 않았다.**
- 하드코딩 색·px 0, 토큰 100%.
- 환자 화면 비노출 원칙(위험신호·메모·RRN·연락처 원문) 위반 0.
- 발견된 불일치·백로그는 별도 목록으로 정리되어 사용자에게 전달됨.

시작 전에 위 **절대 규칙**(특히 1·2·6·7)을 다시 확인하고, Phase 0부터 진행한다.
