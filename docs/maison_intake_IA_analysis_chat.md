# Maison de Balance · 초진 설문 정보구조(IA) 설계서

> 대상 시스템: Next.js + Vercel + Supabase + Admin Dashboard + PDF 출력 + (추후) 건강점수
> 본 문서 범위: 설문 정보 구조화만 수행. 디자인·코드·SQL은 포함하지 않음.
> 원본: Google Form 23페이지 / 원문 문항 의미는 변경하지 않고, 표현만 부드럽게 다듬음.

---

## 범례 (표에서 공통 사용)

**응답 타입**
`short_text` · `long_text` · `single_choice` · `multiple_choice` · `number` · `time` · `date` · `scale(grid)` · `yes_no`

**정보 분류**
- `일반` : 비민감 일반정보
- `개인` : 개인식별정보(PII)
- `고유` : 고유식별정보(주민등록번호) — 최고 등급
- `건강` : 건강정보
- `민감` : 개인정보보호법상 민감정보(건강·성생활 등)

**활용 플래그**
- `D` : Dashboard 목록/상단에 핵심 노출
- `P` : PDF(상담 준비용) 출력 포함
- `S` : 건강점수 계산 후보
- `!` : 상담 전 상담자가 반드시 확인할 위험·주의 신호

> ※ 원문 문항과 "환자 노출 문구"는 원본 폼에서 이미 동일하게 정제되어 있어, 표에는 환자 노출 문구를 싣고 **개선이 필요한 경우에만 표 아래 주석으로 표기**했습니다.

## A. 설문 전체 구조 요약

- **총 13개 논리 섹션 / 약 63개 문항** (원본 12개 섹션 + 신체 계측을 독립 섹션으로 분리).
- 성격상 4개 묶음으로 나뉩니다.
  1. **신원·동의** (SEC 00–01) : 행정·법적 기반. 가장 민감(주민번호).
  2. **목표·맥락** (SEC 02–04) : 상담의 중심. 환자의 동기/기대/체중 이력.
  3. **생활 리듬** (SEC 05–08) : 수면·식사·음주카페인·수분. 한의학적 진단의 핵심 데이터, 문항 수가 가장 많음.
  4. **몸의 신호·임상** (SEC 09–13) : 신체 신호·여성 건강·질환/복약·가족력·자유서술. 임상 안전성과 직결.
- **필수 문항은 7개뿐**(성함·주민번호·연락처·성별·동의·방문 이유·3개월 후 목표). 나머지는 선택 → 환자 이탈을 줄이는 구조로 유지 권장.
- **조건부 문항이 다수** 존재(여성 건강, 다이어트 방법, 붓는 부위, 소화 증상, 알레르기/수술/운동/가족력 상세). 온라인에서는 **분기 로직(show/hide)** 으로 처리해야 모바일 작성 부담이 크게 줄어듭니다.
- 입력 순서(환자 친화)와 열람 순서(상담자 친화)는 **분리 설계**가 바람직합니다(상세는 E 참조).

## B. 추천 섹션 구성

브랜드의 `No.01–No.12` 넘버링 컨벤션(Balance No.1)과의 연속성을 유지하되, 흐름을 개선했습니다.

| 섹션 | 명칭 | No. 매핑 | 성격 | 비고 |
|---|---|---|---|---|
| SEC 00 | 기본 정보 | No.00 | 신원·연락 | 주민번호 포함(고유식별정보) |
| SEC 01 | 개인정보 수집·활용 동의 | No.01 | 동의 | **건강·민감정보 동의 범위 확장 필요(H 참조)** |
| SEC 02 | 방문 동기와 목표 | No.02 | 핵심 | 설문의 중심, 필수 2문항 |
| SEC 03 | 신체 계측·체중 이력 | No.03 | 정량 | **원본에서 분리·전진 배치 권장** |
| SEC 04 | 다이어트 이력 | No.04 | 서술 | 03 다음으로 이동 |
| SEC 05 | 수면 리듬 | No.05 | 생활 | 문항 多, 환경요인 포함 |
| SEC 06 | 식사 리듬 | No.06 | 생활 | grid 포함 |
| SEC 07 | 음주·카페인 | No.07 | 생활 | **원본에선 식사 섹션에 섞여 있던 항목을 독립화** |
| SEC 08 | 수분·미네랄 리듬 | No.08 | 생활 | |
| SEC 09 | 몸의 신호 | No.09 | 임상 | 조건부 분기 多 |
| SEC 10 | 여성 건강 리듬 | No.10 | 임상·민감 | **성별=여성일 때만 노출(자동 분기)** |
| SEC 11 | 건강 이력·생활 패턴 | No.11 | 임상 | 질환·복약·알레르기·직업·운동 통합 |
| SEC 12 | 가족력 | No.12 | 임상 | |
| SEC 13 | 마지막 질문 | No.13 | 정성 | 자유 서술 |

**원본 대비 핵심 변경 3가지**
1. **신체 계측(체중·키·목표체중 등)을 독립 섹션으로 분리하고 다이어트 이력보다 앞에 배치** — 빠르게 답할 수 있는 정량 문항을 먼저 두어 인지 부담을 낮추고, 서술형 회고(다이어트 이력)는 그 다음으로.
2. **음주·카페인을 식사 리듬에서 분리** — 데이터 성격이 다르고(대사·수면 영향), Dashboard 필터링·점수화 단위가 달라 독립 관리가 유리.
3. **여성 건강 섹션을 명시적 분기 처리** — 원본의 "남성은 건너뛰셔도 됩니다" 안내문 대신, 성별 응답에 따라 섹션 자체를 자동 노출/비노출.

## D. Supabase 저장 관점 정리

> SQL은 작성하지 않으며, 데이터가 "어디에 어떤 형태로 들어가야 하는가"의 관점만 정리합니다.

### D-1. patients (환자 1인 = 1행) — 신원·행정 기본
- 00-01 성함, 00-02 주민등록번호(**암호화 저장**), 00-03 연락처, 00-04 주소, 00-05 성별, 00-06 출산 경험, 00-07 내원 경로, 00-08 소개자
- 01-01 동의 여부 + 동의 시각(타임스탬프) + 동의 버전
- 파생: 생년월일·연령(주민번호에서 분리 저장 권장), 성별 식별자

### D-2. survey_responses (제출 1건 = 1행) — 응답 본문
- `patient_id`(FK), `submitted_at`, `status`(신규/검토중/상담완료), `consultant_memo`, `pdf_generated`(bool)
- `raw_answers` **JSONB** : SEC 02~13 전체 문항을 `{문항ID: 값}` 구조로 원형 저장(특히 자유서술·다중선택·grid·조건부 항목)

### D-3. Dashboard용 별도 컬럼화(비정규화) 권장 — 빠른 필터·정렬용
JSONB에서 자주 조회·정렬하는 값을 컬럼으로 승격:
- `primary_goal_text`(02-01 요약/원문), `chief_complaints`(02-03 배열), `current_weight`(03-01), `height`(03-02), `target_weight`(03-03), `bmi`(파생), `exercises_regularly`(11-07)
- `risk_flags`(배열, `!` 규칙으로 산출 — I-4 참조)
- (추후) `health_score`(정수)

### D-4. 추후 건강점수 계산에 필요한 정보
- 정량: 03-01~04, 05-01~03, 06-04, 07-01·05, 11-07~08
- 정성→점수화 후보: 05-04, 06-05~10, 08-01~02, 09-01~08, 10-01
- 별도 `score_components` JSONB(영역별 부분점수) 보관 권장 → 재계산·설명가능성 확보.

### D-5. PDF 출력 전용으로만 필요한 정보
- 수면 환경 상세(05-05~05-10), 음주 상세(07-02~04), 식사 시간 4종(06-01~04) 등 **Dashboard 목록엔 불필요하나 상담 출력엔 유용한** 항목 → JSONB에만 두고 PDF 렌더 시 조회.

## E. Dashboard 표시 구조

### E-1. 응답 목록(리스트) 핵심 컬럼
환자명(마스킹 옵션) · 연락처(마스킹) · 성별 · 제출일 · 핵심 목표(02-01 요약) · 주요 불편(02-03 칩) · **위험신호 뱃지(risk_flags 개수)** · BMI · 상태값 · 메모 유무 · PDF 출력 여부

- 정렬/필터: 제출일, 상태값, 위험신호 유무, 성별, BMI 구간, 주요 불편 칩.
- 위험신호가 있는 행은 상단/강조 처리(브랜드 톤 유지: 빨강 대신 차분한 앰버/버건디 포인트 권장).

### E-2. 개별 응답 상세 — 상담자 친화 열람 순서 (입력 순서와 분리)
1. **신원·연락 요약 + 동의 상태**
2. **위험신호 요약 배너**(`!` 항목 한눈에)
3. **방문 동기·목표**(SEC 02) — 가장 먼저 읽을 핵심
4. **신체 계측·BMI·체중 이력**(SEC 03)
5. **건강 이력·복약·알레르기**(SEC 11) — 임상 안전
6. **가족력**(SEC 12)
7. **여성 건강**(SEC 10) — 해당 시
8. **생활 리듬**: 수면(05) / 식사(06) / 음주·카페인(07) / 수분(08)

> 근거: 상담자는 "위험 → 목표 → 임상 이력"을 먼저 확인한 뒤 생활 데이터로 들어가는 흐름이 효율적. 환자 입력 순서(B)와 다르게 재배열하는 것이 핵심.

## F. PDF 출력 구조

- 용도: **상담 직전 1~2페이지 임상 요약지**. 환자 입력 순서가 아닌 상담자 열람 순서(E-2)를 따름.
- 브랜드 적용: 크림 배경(#F5F0E6)·딥 차콜 본문(#1D1B17)·샴페인 골드(#B29355) 얇은 헤어라인, 향수 라벨식 `No.` 섹션 번호. (※ 본 단계에서 실제 디자인/렌더는 진행하지 않음 — 구조만 정의)
- 구성:
  1. 헤더: 환자명 · 성별/연령 · 제출일 · **주민번호 마스킹 표기(앞 6자리 + 뒷자리 ***)**
  2. **위험신호 요약 박스**(`!` 항목)
  3. 방문 목표(02) → 계측·BMI·체중 이력(03)
  4. 건강 이력·복약·알레르기(11) / 가족력(12) / 여성 건강(10, 해당 시)
  5. 생활 리듬 요약(05~08) — 표/칩 형태로 압축
  6. 몸의 신호(09)
  7. 다이어트 이력(04) · 자유 서술(13)
- 출력 시 주의: **전체 주민번호 미출력**, 출력 이력 로깅, 인쇄물 보관·파기 정책 연계(H 참조).

- 건강정보: 신체 계측, 생활 리듬, 몸의 신호 등 대부분의 설문
- **민감정보(특별 취급)**: 진단 질환(11-01), 복약(11-03), 수술(11-02), 여성 건강(10-01), 가족력 상세(12-02)
- **고유식별정보(최고 등급)**: 주민등록번호(00-02)

**핵심 권고**
1. **주민번호(00-02)**: 가능하면 전체 수집을 피하고 본인확인에 필요한 최소 정보만 사용 검토. 수집 시 **암호화 저장 필수**, UI·PDF에서는 항상 마스킹, JSONB 평문 저장 금지, 접근 로그 기록.
2. **동의 범위 보완(중요)**: 현재 동의 본문이 성명·연락처·생년월일·주소만 명시 → **건강정보(민감정보) 수집·이용에 대한 별도 동의 항목 추가** 필요. 동의 시각·버전 저장.
3. **Dashboard 마스킹**: 주민번호·연락처는 기본 마스킹, 권한 있는 상담자만 일시 해제. 목록 화면에는 민감 질환명 직접 노출 최소화(칩/뱃지로 추상화).
4. **PDF 주의**: 전체 주민번호 미출력, 출력 이력 로깅, 인쇄물 보관·파기 절차 연계.
5. **접근통제(RLS)**: 익명 사용자는 제출(INSERT)만 가능, 조회(SELECT)·수정은 인증된 관리자만. 환자 응답 간 상호 조회 차단.
6. **절대 비노출**: 다른 환자의 응답, 원본 주민번호, 관리자 메모는 환자/외부에 노출 금지.

## I. Claude Code 구현 준비 요약

다음 단계(Next.js + Supabase 구현)에서 Claude Code에 전달할 핵심:

**I-1. 스택·구조**
- Next.js(App Router) + Supabase(Postgres·Auth·RLS) + Vercel.
- 테이블 2개: `patients`(신원), `survey_responses`(응답 본문 + 비정규화 컬럼). 문항 정의는 우선 코드에 상수로 고정(추후 `questions` 테이블로 동적화 가능).

**I-2. 필드·타입 명세**
- 본 문서 **C(문항표)** 의 ID·타입·필수·분류·플래그를 단일 소스로 사용. D 섹션의 저장 위치 매핑(patients / JSONB / 비정규화 컬럼) 적용.

**I-3. 검증 규칙**
- 필수 7문항(00-01·00-02·00-03·00-05·01-01·02-01·02-02).
- 주민번호: 숫자 13자리·하이픈 제외. 연락처: 숫자 형식. 체중·키·목표체중·음주·커피: 양수, 상식 범위 가드. 시간 항목: HH:MM.

**I-4. 조건부(분기) 로직 목록**
- 00-08 ← 00-07='지인 소개'
- 03-05·03-06 ← 00-06≠'없음'
- 04-02 ← 04-01='있음'
- 09-04 ← 09-03≠'거의 붓지 않는다'
- 09-06 ← 09-05='잘 안 됨'
- SEC 10 전체 ← 00-05='여성'
- 11-02 ← 수술 이력 있음 / 11-05 ← 11-04='있음' / 11-08 ← 11-07='있음'
- 12-02 ← 12-01='있음'

**I-5. 위험신호(risk_flags) 산출 규칙(초안)**
- 11-01에 당뇨·고혈압·심혈관·갑상선·신장 포함 / 11-03에 항우울제·스테로이드·피임약 포함 / 11-04='있음' / 05-04='거의 날을 새움' / 06-06에 '폭식 후 죄책감' / 06-10='입맛 떨어짐' / 08-02='주황·갈색' 또는 '거품' / 09-01에 '컨디션 저하' / 09-04='전신' / 목표 BMI 저체중 구간 → 각 항목 발생 시 `risk_flags`에 추가.
- 단, **자동 경고 문구를 환자에게 노출하지 않음**. 상담자 검토 큐로만 사용.

**I-6. UX·스텝 구조**
- 모바일 우선 위저드: 1스텝 = 1섹션(SEC 09·11은 문항이 많아 하위 분할 권장). 스티키 진행바, 필수 미입력 인라인 검증, 조건부 문항 즉시 show/hide.
- 동의(SEC 01)는 신원 직후 배치, 건강정보 동의 별도 체크 포함.

**I-7. 파생값**
- BMI(03-01·03-02), 목표 BMI, 수면시간(05-01~02), 체중 격차(현재–최저/현재–목표) → 앱 계산 또는 DB 계산 후 비정규화 컬럼 반영.

**I-8. 이번 단계에서 하지 않을 것**
- 최종 점수식 확정, 최종 비주얼 디자인, 환자용 자동 진단/경고 문구.
