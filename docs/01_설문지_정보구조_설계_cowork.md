# Maison de Balance · 초진 설문 — 정보 구조 설계서

> 대상: 압구정 프리미엄 한방 다이어트·웰니스 의원 환자 사전 상담 설문
> 목적: Google Form 원문을 Next.js + Supabase + Dashboard + PDF + 건강점수 계산까지 확장 가능한 정보 구조로 재정리
> 범위: **정보 구조(IA) 설계만 수행.** 디자인·코드·SQL은 작성하지 않음. 문항 의미와 한방 사전상담 맥락은 그대로 유지.

---

## A. 설문 전체 구조 요약

원문 설문(Google Form, 23개 화면 캡처)은 다음과 같은 흐름을 가진다.

- 총 **13개 화면 섹션**, 약 **65개 문항**으로 구성된다.
- 문항 성격은 크게 다섯 갈래다: ① 환자 식별/연락 정보, ② 동기·목표(상담 핵심), ③ 생활 리듬 데이터(수면·식사·수분·운동), ④ 신체 신호·증상 데이터, ⑤ 의료 이력·가족력(민감 건강정보).
- 원문은 한방의 "균형(밸런스)" 관점을 따라 **수면 → 식사 → 수분 → 몸의 신호 → 여성 건강** 순으로 몸의 리듬을 훑는 구조다. 이 흐름 자체는 임상적으로 타당하므로 큰 골격은 유지한다.
- 다만 온라인 설문 관점에서 두 가지를 조정한다: (1) **주민등록번호 전체 수집**은 초진 단계에서 과도하므로 재검토 대상, (2) 민감 의료 이력을 설문 후반부로 두어 환자의 초기 이탈을 줄이는 현재 순서는 그대로 유지하는 것이 좋다.

핵심 설계 원칙은 다음과 같다.

1. **환자가 직접 입력하는 식별 정보(patients 테이블)** 와 **설문 응답(survey_responses)** 을 물리적으로 분리한다.
2. 상담자가 0~30초 안에 판단하는 핵심 값(상담 목표·주요 증상·위험 신호·BMI 등)은 별도 컬럼으로 승격(promote)한다.
3. 나머지 상세 응답은 `raw_answers` JSONB에 원형 그대로 보존하여 문항 변경에도 스키마가 깨지지 않게 한다.
4. 모든 문항은 **문항 ID**를 부여하여 PDF·점수화·Dashboard가 동일 키를 참조하도록 한다.

## B. 추천 섹션 구성

원문 13개 화면을 온라인 멀티스텝(step) 기준으로 **10개 논리 섹션**으로 재편한다. 괄호 안은 원문 위치.

| 순서 | 섹션 코드 | 섹션명 | 목적 | 원문 화면 |
|---|---|---|---|---|
| 0 | `consent` | 개인정보 수집·활용 동의 | 법적 동의 선취득. 가장 먼저 배치 권장 | p.3 |
| 1 | `basic` | 기본 정보 | 환자 식별·연락·본인확인 | p.1–2 |
| 2 | `goal` | 몸의 목표와 기대 | **상담 핵심.** 방문 동기·기대·주관심 영역 | p.4 |
| 3 | `body_metrics` | 체형·다이어트 이력 | 체중/키/목표 및 과거 시도 맥락 | p.5–6 |
| 4 | `sleep` | 수면 리듬 | 대사·호르몬 기반 데이터 | p.7–9 |
| 5 | `meal` | 식사 리듬 (음주·카페인 포함) | 식습관·기호식품 패턴 | p.10–13 |
| 6 | `hydration` | 수분 & 미네랄 리듬 | 수분 섭취·신장 부담 단서 | p.14 |
| 7 | `body_signal` | 몸의 신호 | 체온·부종·소화·배변 등 한의학적 변증 단서 | p.15–17 |
| 8 | `female` | 여성 건강 리듬 | 생리 주기 관련 (여성 전용, 조건부) | p.18 |
| 9 | `medical` | 건강 이력·복약·알레르기 | **민감 의료정보.** 안전성 판단 | p.19–20 |
| 10 | `lifestyle` | 직업·운동·가족력 | 활동량·유전적 소인 | p.21–22 |
| 11 | `final` | 마지막 질문 | 자유 서술, 상담 라포 형성 | p.23 |

> 동의(`consent`)를 0번으로 끌어올린 점만 원문과 다르다. 동의 전 식별정보를 입력받는 구조는 개인정보 수집 동의의 선후 관계상 부적절할 수 있으므로, **동의 → 기본정보** 순서를 권장한다.

**환자 작성 흐름(권장):** 동의 → 가벼운 식별정보 → "왜 오셨나요"(동기, 몰입 유도) → 체형 → 생활 리듬(수면·식사·수분) → 몸의 신호 → 여성 건강 → 의료 이력(민감, 후반) → 생활/가족력 → 자유 서술(마무리).

**상담자 열람 흐름(권장):** 위험 신호/의료 이력 → 상담 목표 → 체형·BMI → 생활 리듬 요약 → 상세. (Dashboard 상세 페이지 순서는 E 참조)

## C. 문항별 구조화 표

표기 약속:
- **응답타입**: `short_text` / `long_text` / `single_choice` / `multiple_choice` / `number` / `date` / `time` / `scale` / `yes_no`
- **민감도**: 일반PI(일반 개인정보) / 건강(일반 건강정보) / 민감(민감 건강정보)
- **저장위치**: `patients`(컬럼) / `promote`(survey_responses 승격 컬럼) / `JSONB`(raw_answers)
- **D**: Dashboard 목록 노출, **PDF**: 출력 포함, **점수**: 건강점수 후보, **⚠️**: 상담자 위험신호 확인 항목

원문 설문은 `time`을 별도 타입으로 제공하지 않고 자유 입력이므로, 시간 문항은 `time`(권장) 또는 `short_text`로 표기한다. 원문에 없던 타입을 강제하지 않되, 데이터화 시 권장 타입을 병기한다.

## D. Supabase 저장 관점 정리

> SQL은 작성하지 않는다. 데이터 모델링에 필요한 분류 관점만 제시한다.

### D-1. `patients` 테이블 (환자 마스터 — 1인 1행, 재방문 시 갱신)

환자를 식별·연락하는 안정적 속성. 설문 응답과 분리해 중복 입력을 피하고 재방문 환자에 연결한다.

- `basic_name`, `basic_phone`, `basic_address`, `basic_sex`
- `basic_rrn` — **암호화 저장**(또는 생년월일+성별로 대체). 평문 저장 금지.
- `consent_agree`, 동의 일시(`consent_at`)
- 시스템 컬럼: `patient_id`(PK), `created_at`, `updated_at`

### D-2. `survey_responses` 테이블 (제출 1건 = 1행)

- `response_id`(PK), `patient_id`(FK), `submitted_at`, `status`(아래 E-2), `admin_memo`, `pdf_generated`(bool)
- `raw_answers` **JSONB** — 모든 문항의 원형 응답을 `{문항ID: 값}` 형태로 통째 보존. 문항 추가·삭제·문구 변경에도 스키마 마이그레이션 불필요.

### D-3. 승격(promote) 컬럼 — Dashboard 필터·정렬·점수 계산용

`raw_answers`에도 있지만, 빠른 조회를 위해 별도 컬럼/생성열로 복제 또는 추출하면 좋은 값:

- 식별/요약: `goal_reason`(요약), `goal_concern_areas`(배열), `final_message`(요약)
- 신체 지표: `body_weight_current`, `body_height`, `body_weight_target`, **`bmi`(계산 파생열)**
- 위험·안전: `medical_conditions`, `medical_medications`, `medical_food_allergy`, `medical_allergy_detail`, `family_history`, `sleep_aid_use`
- 점수 입력값: `sleep_quality`, `sleep_latency_min`, `meal_regularity`, `meal_habits`, `meal_appetite`, `signal_edema`, `signal_digestion`, `signal_bowel_freq`, `urine_color`, `hydration_pattern`, `signal_temp_energy`, `female_menstrual`, `life_exercise`, `alcohol_freq_week`
- 분류: `basic_referral`, `basic_sex`, `life_occupation`

### D-4. JSONB 전용 (상세 보존, 빠른 조회 불필요)

수면 환경 세부(`sleep_temp` 등), 식사 시간대, 음주·카페인 세부, 부종 부위, 소화 증상, 다이어트 이력 서술 등 — 상담 상세 페이지·PDF에서만 사용.

### D-5. PDF 전용 데이터

별도 저장은 불필요. PDF는 `patients` + `survey_responses.raw_answers`를 문항 ID 기준으로 렌더링하면 된다. 단, **마스킹 정책**(주민등록번호 등)을 PDF 생성 로직에서 분기 처리할 수 있도록 민감 플래그를 메타로 들고 있을 것을 권장.

| BMI/체중 | `bmi`, `body_weight_current` | |
| 위험 신호 | `medical_conditions`·`medical_food_allergy`·`sleep_aid_use` 등 → 배지 | ⚠️ 배지 |
| 상태값 | `status` | 신규/검토중/상담완료 |
| 관리자 메모 | `admin_memo` | 1줄 |
| PDF | `pdf_generated` | 출력 여부 아이콘 |

### E-2. 상태값(`status`) 권장 enum

`신규 제출` → `검토 중` → `상담 예정` → `상담 완료` → `보류/취소`

### E-3. 개별 응답 상세 페이지 — 권장 표시 순서

상담자 의사결정 순서(위험→목표→객관지표→리듬→상세)에 맞춘다.

1. **상단 고정 요약 바**: 환자명·성별·나이·연락처(마스킹), BMI, `status`, PDF 버튼
2. **⚠️ 안전·위험 신호**: 진단 질환, 복용 약, 알레르기, 가족력, 수면제 복용, 무리한 목표 체중 — 가장 먼저
3. **상담 핵심**: `goal_reason`, `goal_3month`, `goal_concern_areas`, `final_message`
4. **체형·다이어트 이력**: 체중 지표, BMI, 다이어트 시도/방법, 신경 쓰이는 부위
5. **생활 리듬**: 수면 → 식사(음주·카페인) → 수분 (점수 요약 카드와 함께)
6. **몸의 신호**: 체온·부종·소화·배변
7. **여성 건강**(여성만)
8. **직업·운동·가족력**
9. **원문 전체(raw) 보기 토글** — 누락 없이 확인용
10. **관리자 메모 입력 / 상태 변경 / PDF 출력** 액션

## F. PDF 출력 구조

상담실/차트 보관용. 상담 상세 페이지(E-3)와 동일한 논리 순서를 따르되, 인쇄 친화적으로 구성.

- **표지/헤더**: 의원명(Maison de Balance), "초진 상담지", 환자명, 제출일, 본인확인 정보(마스킹 적용)
- **1면 — 요약 시트**: 기본정보, BMI·체중 지표, 상담 목표, 주요 관심 영역, ⚠️ 위험 신호 요약, (선택) 건강점수 요약
- **2면 이후 — 섹션 상세**: 수면 / 식사·음주·카페인 / 수분 / 몸의 신호 / 여성 건강 / 건강 이력 / 생활·가족력 순으로 문항-응답 나열
- **마지막 — 자유 서술**(`goal_*` 서술형, `final_message`)과 상담자 메모란(수기 작성 공백)
- **출력 정책**:
  - 주민등록번호는 마스킹(예: `901011-1******`) 또는 미출력 기본값.
  - 민감 의료정보는 출력되지만 문서 자체가 의료정보이므로 **출력물 취급·보관 주의 문구**를 푸터에 명시 권장.
  - 미응답 문항은 "—"로 표기하여 빈칸과 누락을 구분.

## G. 건강점수 계산 후보 항목

> 점수식은 아직 만들지 않는다. 점수화 적합성과 주의점만 표시.

| 영역 | 후보 문항ID | 점수화 가능성 | 주의점 |
|---|---|---|---|
| 수면 | `sleep_quality`, `sleep_latency_min`, `sleep_bedtime`/`waketime`(수면시간 파생), `sleep_aid_use` | 높음 | 다중선택을 점수로 환산하려면 항목별 가중치 사전 정의 필요. 취침/기상은 자유입력이라 파싱 정규화 선행. |
| 식욕·식습관 | `meal_appetite`, `meal_habits`, `meal_skip_reaction`, `meal_speed` | 중간 | 주관적 표현. '폭식 후 죄책감' 등은 섭식 문제 신호일 수 있어 점수보다 ⚠️ 플래그가 적절할 수 있음. |
| 식사 규칙성 | `meal_regularity`(grid), `meal_time_*` | 높음 | grid는 끼니별 이진값이라 점수화 용이. |
| 소화 | `signal_digestion`, `signal_digestion_symptoms`, `signal_bowel_freq`, `signal_stool` | 높음 | 증상 개수·빈도 기반 척도화 적합. |
| 부종 | `signal_edema`, `signal_edema_area` | 높음 | 3단계 빈도라 정량화 쉬움. |
| 피로·에너지 | `signal_temp_energy`(쉽게 지침/컨디션 저하 등) | 중간 | 다중선택 → 항목 가중치 필요. |
| 생리주기(여성) | `female_menstrual` | 중간 | 여성만 응답 → 점수 비교 시 성별 분리 필수. 결측 처리 정책 필요. |
| 체중 변화 | `body_weight_current`/`min`/`target`, `bmi` | 높음 | BMI는 표준식으로 즉시 계산. 단, 목표가 저체중이면 점수와 별개로 ⚠️. |
| 운동량 | `life_exercise`, `life_exercise_detail` | 중간 | 상세가 자유서술이라 횟수·시간 구조화 입력으로 바꾸면 점수화 정확도↑. |
| 스트레스 | (직접 문항 없음) `sleep_quality`의 '걱정거리 뒤척임', `final_message` | 낮음 | 전용 문항이 없어 간접 추정만 가능. 추후 스트레스 척도 문항 추가 검토. |
| 복약/기호식품 | `sleep_aid_use`, `alcohol_freq_week`/`amount`, `caffeine_cups_day`/`reaction` | 중간 | 음주량 자유입력 정규화 필요. 점수보다 위험 보정 인자로 활용 권장. |
| 수분 | `hydration_pattern`, `urine_color` | 중간 | 소변 색은 자가보고라 신뢰도 한계. |

공통 주의: ① 다중선택·자유서술은 점수화 전 **항목 사전·가중치 정의**가 선행되어야 함. ② 점수는 진단이 아니라 상담 참고 지표임을 명시. ③ 결측(미응답)·조건부 문항(여성/출산)의 처리 규칙을 점수식과 함께 설계.

## H. 민감정보 / 보안 주의사항

> 법률 자문이 아니라 시스템 설계상 권고 사항이다. 정확한 법적 요건은 개인정보보호 전문가/변호사 검토를 권한다.

**정보 분류**

- **일반 개인정보**: 성함, 연락처, 주소, 성별, 내원 경로, 소개자, 직업
- **일반 건강정보**: 체중·키·BMI, 수면·식사·수분·운동 리듬, 몸의 신호(부종·소화·배변 등), 목표·기대
- **민감 건강정보(특히 보호)**: 진단 질환력, 수술 이력, 복용 약(항우울제·스테로이드·피임약 등), 알레르기, 생리·여성 건강, 가족력, 수면제·안정제 복용
- **고유식별정보**: 주민등록번호 — 가장 강한 보호 대상. 수집 최소화 원칙상 **생년월일+성별 대체** 우선 검토.

**Dashboard 마스킹 권장**

- 주민등록번호: 목록·상세 모두 기본 마스킹, 권한 있는 관리자만 토글 노출.
- 연락처: 목록에서 부분 마스킹(`010-****-1234`).
- 민감 의료정보: 목록에서는 "배지/플래그"만, 구체 내용은 상세 페이지에서만.

**PDF 출력 주의**

- 주민등록번호는 마스킹 또는 미출력 기본.
- 출력물은 그 자체로 민감 의료문서 → 취급·파기 주의 문구, 인쇄 이력 관리 권장.

**관리자 외 절대 비노출**

- 주민등록번호, 진단 질환·복약·알레르기·가족력 등 모든 민감 건강정보, 자유서술(`final_message`·상담 동기).

**시스템 설계 권고**

- Supabase **RLS로 행 단위 접근 통제** — 인증된 관리자(staff) 역할만 응답 조회.
- 민감 컬럼(주민번호 등) 저장 시 암호화, 키 분리 보관.
- 동의(`consent_agree`)와 동의 일시를 반드시 기록하고, 동의 없는 제출은 차단.
- 접근 로그(누가 언제 어떤 응답을 열람·출력했는지) 적재 권장.
- 데이터 보관 기간·파기 정책 사전 정의.

## I. 다음 단계: Claude Code 전달용 구현 준비 요약

다음 단계(구현)에서 Claude Code에 넘길 수 있는 명세 요약. **여기서 SQL·코드는 만들지 않았고, 아래는 구현 입력으로 쓸 정리다.**

**1. 데이터 모델 골격(구현 시 확정)**
- `patients` (환자 마스터): 식별·연락·동의. 주민번호는 암호화 또는 생년월일+성별 대체.
- `survey_responses`: `patient_id` FK + `raw_answers` JSONB + 승격 컬럼(D-3) + `status`/`admin_memo`/`pdf_generated`.
- `bmi`는 체중·키 기반 파생열로 계산.

**2. 문항 정의는 단일 소스로 관리**
- 본 문서 C의 문항 표를 **설문 정의 JSON/상수 파일**로 만들어 프론트 폼·검증·PDF·점수 계산이 같은 문항 ID를 참조하게 한다. (문항 추가/수정이 한 곳에서 끝나도록)
- 각 문항 메타: `id`, `section`, `label`, `type`, `options`, `required`, `sensitivity`, `conditional`(분기 조건), `promote`(여부), `pdf`(여부), `score`(여부), `riskFlag`(여부).

**3. 조건부 분기 로직(구현 필요)**
- `basic_sex = 남성` → `female` 섹션 건너뜀
- `diet_tried = 있음` → `diet_methods` 노출
- `signal_edema ∈ {자주, 가끔}` → `signal_edema_area` 노출
- `signal_digestion = 소화 안 됨` → `signal_digestion_symptoms` 노출
- `medical_food_allergy = 있음` → `medical_allergy_detail` 노출
- `life_exercise = 있음` → `life_exercise_detail` 노출
- `family_history = 있음` → `family_history_detail` 노출
- `basic_referral = 지인 소개` → `basic_referrer` 노출
- 출산 관련 체중 문항은 `basic_birth_count ≠ 없음`일 때만 노출 권장

**4. UX/멀티스텝**
- 멀티스텝(섹션당 1스텝, 진행률 표시) 권장. 모바일 우선.
- 동의를 첫 스텝으로, 민감 의료 이력은 후반 스텝으로.
- 필수는 최소화(식별 4개·동의·상담 동기 2개 정도), 나머지는 선택으로 두어 완료율↑.
- 자유서술 placeholder는 원문 예시 문구를 그대로 사용.

**5. Dashboard/PDF/점수**
- Dashboard 목록·상세 구조는 E, PDF 구조는 F, 점수 후보·주의점은 G를 사양으로 사용.
- 점수식은 별도 단계에서 항목 가중치 정의 후 설계.

**6. 보안**
- RLS, 민감 컬럼 암호화, 동의 기록, 접근/출력 로그, 마스킹 정책(H)을 구현 체크리스트로 포함.

### 부록 · 온라인화 시 검토 권장 사항(원문 대비 변경 후보)

원문의 의미는 유지하되, 데이터 품질·완료율을 위해 구현 단계에서 검토할 만한 항목:

1. 주민등록번호 → 생년월일+성별 대체(고유식별정보 최소 수집).
2. 시간 입력 문항(취침/기상/식사)은 자유 텍스트 대신 `time` 입력으로 → 점수화·파싱 정확도↑.
3. `life_exercise_detail`, `alcohol_amount` 등 자유서술을 구조화 입력(횟수/시간/단위 분리)으로 → 점수화 용이.
4. 현재 체중·키는 BMI·점수의 기반이므로 **필수**로 승격 권장.
5. 출산 관련 문항은 성별/출산 경험에 따라 조건부 노출하여 불필요한 노출 최소화.
